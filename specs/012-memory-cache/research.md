# Research: Thread-Safe In-Memory Cache

**Feature**: `012-memory-cache`
**Date**: 2026-03-02

## R1: LRU Cache Library Selection

**Decision**: Use `hashicorp/golang-lru/v2/expirable`

**Rationale**:

- Provides generic LRU (`[K, V]`) with built-in TTL expiry
- Thread-safe via internal locking — no manual `sync.RWMutex`
- Battle-tested (5k+ GitHub stars, used across HashiCorp
  ecosystem)
- v2.0.7 is latest stable (Sep 2023), no known issues
- API surface is minimal: `NewLRU`, `Add`, `Get`, `Len`,
  `Remove`, `Purge`

**Alternatives considered**:

- **Hand-rolled `sync.RWMutex` + `container/list`**: More
  code to write and maintain, higher risk of subtle
  concurrency bugs. No advantage over a well-tested library.
- **`dgraph-io/ristretto`**: More features (admission
  policy, metrics) but heavier dependency, Apache 2.0
  license. Overkill for a simple key-value pricing cache.
- **`patrickmn/go-cache`**: Simpler but no LRU eviction,
  only TTL-based expiry. Doesn't satisfy FR-006 (LRU
  eviction at max size).
- **`allegro/bigcache`**: Designed for large datasets with
  GC optimization. Stores `[]byte` values, requiring
  serialization. Unnecessary complexity for `[]PriceItem`.

## R2: License Compatibility

**Decision**: MPL 2.0 is compatible with Apache 2.0 project

**Rationale**:

- MPL 2.0 is "weak copyleft" — requires modifications to
  MPL-licensed files to stay under MPL, but allows combining
  with Apache 2.0 at project level
- The project already depends on `hashicorp/go-retryablehttp`
  (MPL 2.0), establishing precedent
- No file-level modifications needed; the library is used
  as-is via its public API

**Alternatives considered**: None — all HashiCorp Go libraries
use MPL 2.0 and the compatibility is well-established.

## R3: finfocus-spec Dependency Bump

**Decision**: Bump from v0.5.4 to v0.5.7

**Rationale**:

- v0.5.7 adds `expires_at` field on `ActualCostResult`
  (field 8) and `GetProjectedCostResponse` (field 13)
- v0.5.7 adds `BatchCost` RPC with `BatchCostRequest` /
  `BatchCostResponse` messages
- SDK provides zero-alloc helpers: `IsActualCostExpired`,
  `ActualCostExpiresAt`, `IsProjectedCostExpired`,
  `ProjectedCostExpiresAt`
- SDK provides `WithProjectedCostExpiresAt` functional option
- Backward compatible — existing plugins work without
  modification

**Alternatives considered**:

- **Stay on v0.5.4**: Would require omitting `expires_at`
  from responses, breaking the L1/L2 cache contract. Not
  viable.
- **Fork/patch v0.5.4**: Unnecessary complexity when the
  upstream version already provides what we need.

## R4: Cache Key Normalization Strategy

**Decision**: Pipe-delimited, lowercase, sorted dimensions

**Rationale**:

- Cache keys are derived from 5 `PriceQuery` dimensions:
  `ArmRegionName`, `ArmSkuName`, `ServiceName`,
  `ProductName`, `CurrencyCode`
- Normalization: lowercase all values, join with `|`
  separator in a fixed canonical order
- Fixed order ensures the same query always produces the
  same key regardless of how it was constructed
- `PriceQuery` fields map directly to `ResourceDescriptor`
  fields in batch operations, enabling key reuse
- Empty fields are included as empty strings to maintain
  position (e.g., `eastus|standard_b1s||virtual machines|usd`)

**Key format**:

```text
{region}|{sku}|{product}|{service}|{currency}
```

**Alternatives considered**:

- **Hash-based keys (SHA256/FNV)**: Compact but
  non-debuggable. Cache misses harder to diagnose in logs.
- **JSON serialization**: Expensive per-lookup and
  ordering-dependent.
- **OData filter string**: Already generated by
  `FilterBuilder` but includes `priceType` and other
  non-query dimensions, producing different keys for the
  same logical query.

## R5: expires_at Computation Strategy

**Decision**: Store absolute expiry time per cache entry;
return it directly on cache hits

**Rationale**:

- `golang-lru/v2/expirable` manages TTL internally but does
  not expose per-entry expiry times
- The cache wrapper must independently track both
  `createdAt` and two expiry values per entry
- Cache value type becomes a wrapper struct holding
  `[]PriceItem`, `createdAt`, and `expiresAt` (for L2)
- Two-TTL model:
  - L1 TTL (24h): governs LRU cache expiry internally
  - `expires_at` TTL (4h): governs the timestamp sent to
    gRPC callers
- On fresh API results:
  `expiresAt = time.Now().Add(expiresAtTTL)`
- On cache hits: return the stored `expiresAt` as-is
  (remaining L2 TTL, per clarification)
- L2 misses re-enter the plugin and are served from the
  still-valid L1 cache without hitting Azure
- Worst case staleness: ~28 hours (24h L1 + 4h L2)

**Alternatives considered**:

- **Single TTL for both L1 and L2**: Simpler but either
  wastes L1 cache value (short TTL) or gives callers
  stale data (long TTL). The two-TTL model balances both.
- **Recompute `expires_at = now + TTL` on every response**:
  Allows L2 to hold data indefinitely via repeated
  refreshes. Rejected during clarification.
- **Expose golang-lru internals**: Not possible — the
  library doesn't expose per-entry metadata.

## R8: Azure Pricing Data Freshness Research

**Decision**: 24-hour L1 TTL is well-justified by Azure data
refresh cadence

**Rationale**:

- Azure Retail Prices API data refreshes **daily**
- Price changes typically take effect on the 1st of month
- `effectiveStartDate` fields show month-boundary dates;
  prices stable for months/years for established SKUs
- Microsoft recommends "query cost data at most once per
  day"
- Industry practice (Infracost): weekly refresh
- Common app-level TTL: 24 hours
- API rate limit: ~10 requests per 60 seconds per client
- API returns `no-cache` headers (prevents proxy caching,
  not a freshness signal)

**Alternatives considered**:

- **1 hour**: Overly aggressive. Azure pricing does not have
  hourly granularity (earlier assumption was incorrect).
  Wastes API calls for no freshness benefit.
- **12 hours**: Reasonable but no advantage over 24 hours
  given daily data refresh.
- **7 days (Infracost model)**: Too long for a real-time
  cost estimation tool; acceptable for static IaC
  analysis but not for interactive CLI queries.

## R6: Observability Requirements

**Decision**: Log cache hit/miss ratio per constitution
Section III

**Rationale**:

- Constitution requires: "Cache hit/miss ratio MUST be logged
  every 1000 requests or 5 minutes (whichever comes first)"
- Implementation: atomic counters for hits and misses,
  checked on each Get call
- Log format: structured JSON via zerolog with fields
  `cache_hits`, `cache_misses`, `cache_hit_ratio`,
  `cache_size`
- Individual cache hits/misses logged at debug level
- Ratio summary logged at info level

**Alternatives considered**:

- **Prometheus metrics**: Not appropriate — plugin is a
  short-lived gRPC process, not a long-running service with
  a metrics endpoint.
- **No observability**: Violates constitution Section III.

## R7: Cache Placement Architecture

**Decision**: Cache wraps `azureclient.Client` as a new
type; `Calculator` receives the wrapped client

**Rationale**:

- The cache sits between `Calculator` and `Client.GetPrices`
- A `CachedClient` type wraps `*Client` and adds cache
  logic, exposing the same `GetPrices` signature
- `Calculator` is modified to accept a `CachedClient` (or
  an interface) instead of calling `Client` directly
- This keeps cache logic in `azureclient` (where `PriceQuery`
  and `PriceItem` live) and keeps `Calculator` focused on
  gRPC concerns (`expires_at` setting, error mapping)
- `Calculator` reads `expiresAt` from the cache response to
  set `expires_at` on gRPC messages

**Alternatives considered**:

- **Cache inside `Calculator`**: Mixes gRPC and caching
  concerns. Makes testing harder.
- **Cache as middleware/interceptor**: Over-engineered for a
  single `GetPrices` method.
- **Cache inside `Client` directly**: Modifies existing
  tested code. The wrapper approach is additive.
